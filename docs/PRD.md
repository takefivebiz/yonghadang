# Corelog PRD (Product Requirements Document)

## 1. 프로젝트 개요 (Project Overview)

### 1.1 브랜드 정의

- **프로젝트명**: Corelog (코어로그)
- **메인 카피**: 사람은, 읽힌다. 나도, 저 사람도, 우리 사이도.
- **서브 카피**: 궁금했던 것들이 이제 보일 거야.
- **브랜드 질문**: 사람 해석을 통해 관계의 구조가 보이면 어떨까?
- **포지셔닝**: 자신만의 패턴이 아닌 "사람 간의 관계 구조"를 이해하는 것

---

### 1.2 제품 포지셔닝

Corelog는 사용자의 선택과 응답 데이터를 기반으로 **사람 간의 관계 구조, 패턴, 상호작용을 이해**하게 하는
**AI 기반 관계 해석 및 질문 탐색 시스템**이다.

**자기해석이 아니라 "사람 해석"**:

- 나의 패턴 (내가 어떻게 생각하고 행동하는지)
- 상대의 패턴 (상대가 어떻게 생각하고 행동하는지)
- 관계의 구조 (우리 사이에서 무엇이 반복되고 있는지)

점술/운세 서비스가 아니라 **Relationship Pattern Recognition System**이다.

## 3. 분석 엔진 설계 (Insight Inference Engine)

### 3.1 설계 원칙 (사람 해석 중심)

- 해석의 근거는 유저가 직접 선택한 데이터여야 한다
- 단일 답변이 아니라 답변 간의 조합과 패턴을 해석한다
- 개인의 선택/패턴뿐 아니라 "사람 간의 상호작용과 관계 구조"를 본다
- 리포트는 상황 설명이 아니라 "관계 구조 기반 상황 해석"이어야 한다
- **분석 타입(나/상대/관계)에 따라 관점이 변경되며, 해석은 일관된다**
- 사용자는 분석 과정을 느끼지 않고, 결과만 자연스럽게 경험해야 한다

---

### 3.2 카테고리 구조 (Category Structure)

Corelog는 다음 4개의 핵심 카테고리로 구성된다.

- 연애
- 인간관계
- 직업/진로
- 감정

각 카테고리는 단순한 주제 분류가 아니라 이후 질문 구조, 해석 방식, 리포트 톤, 확장 질문 생성에
직접적으로 영향을 주는 **1차 해석 컨텍스트**로 작동한다.

#### 카테고리 분리 원칙 (사람 해석 관점)

- **연애**: 감정적 몰입, 상호작용 패턴, 관계의 미래 방향이 중요한 영역  
  → 나의 행동/생각, 상대의 행동/생각, 우리 사이의 반복 구조를 모두 해석
- **인간관계**: 거리감, 역할, 신뢰, 경계 설정이 중요한 영역 (친구, 가족, 직장 등)  
  → 관계 속 자신의 위치, 상대의 성향, 관계의 흐름을 해석
- **직업/진로**: 선택의 기준, 확신도, 타인 vs 자신의 시선 충돌이 중요한 영역  
  → 자신의 욕구/가치, 타인의 기대/평가, 둘 사이의 공백을 해석
- **감정**: 내부 상태, 반복되는 감정 패턴, 감정의 작동 원리가 중요한 영역  
  → 감정 자체의 구조, 상황별 반응 패턴, 감정의 흐름을 해석

#### 하위 분기 구조

| 카테고리  | 하위 분기                                |
| --------- | ---------------------------------------- |
| 연애      | 썸 / 연애 중 / 이별 / 재회               |
| 인간관계  | 친구 / 가족 / 직장 / 사람 전반           |
| 직업/진로 | 이직 / 방향성 / 확신 부족 / 현실 vs 적성 |
| 감정      | 하위 분기 없음 (감정 유형으로 직접 진입) |

---

### 3.3 입력 데이터 구조

질문은 최대 20개까지 확장 가능하며, 다음 데이터를 수집한다.

| 데이터 유형                   | 수집 내용                                             |
| ----------------------------- | ----------------------------------------------------- |
| 현재 상황 (Context)           | 어떤 문제/관계/결정을 고민하는지                      |
| 감정 (Emotion)                | 현재 체감 감정 (불안, 답답함, 회피, 기대감 등)        |
| 가치 기준 (Value Criteria)    | 선택 시 중요 기준 (안정성, 성장, 애정, 신뢰 등)       |
| 행동 반응 (Behavior Pattern)  | 불확실할 때 행동 방식 (미룸, 회피, 타인 의견 따름 등) |
| 반복 구조 (Repeating Pattern) | 비슷한 상황에서 반복되는 선택 방식                    |
| 자기 인식 (Self-Perception)   | 유저가 스스로를 어떻게 이해하는지                     |

#### 입력 수집 원칙

- 모든 질문은 복수 선택 가능하도록 설계할 수 있다
- 초기 버전에서는 자유 입력 기능을 제공하지 않는다
- 사용자는 시스템이 생성한 선택지 기반 질문 세트를 통해 탐색을 진행한다

---

### 3.4 추론 엔진 구조

복수의 답변을 서로 연결하여 **구조적 패턴**을 추론한다.

| 추론 유형             | 설명                                                                 |
| --------------------- | -------------------------------------------------------------------- |
| 성향 추론             | 유저가 어떤 기준으로 판단하는지 (확신형/안정형/탐색형/회피형/검증형) |
| 행동 패턴 추론        | 불확실성, 갈등, 관계, 선택 앞에서 어떻게 움직이는지                  |
| 가치 충돌 추론        | 실제 선택 기준과 마음속으로 원하는 기준 사이의 충돌                  |
| 현재 흐름 추론        | 지금 상태가 어떤 방향으로 흘러가고 있는지                            |
| 질문 확장 포인트 추론 | 리포트 이후 유저가 자연스럽게 궁금해할 지점                          |

#### 추론 원칙

- 동일 의미를 반복 설명하지 않는다
- 유저가 아직 명확히 인식하지 못한 구조를 우선 드러낸다
- 미래 예측보다 **현재 상태의 재해석**을 우선한다

---

### 3.5 무료 리포트 생성 구조

무료 리포트 공통 구조:

1. 유저 상태를 정확히 묘사
2. 유저의 성향/패턴을 짚어줌
3. 반복되는 구조를 드러냄
4. 그 구조가 생기는 작동 원리를 설명
5. 현재 상태가 유지될 경우의 흐름을 제시
6. 핵심 깨달음을 한 문장으로 정리

무료 리포트 원칙:

- "들킨 느낌"을 줘야 한다 (단순 공감 X)
- 해석의 방향을 제시하되, 최종 판단을 확정하지 않는다
- 자연스럽게 다음 질문이 생기는 구조를 만든다

---

### 3.6 질문별 유료 확장 리포트 생성 구조

Corelog의 유료 리포트는 하나의 단일 본편이 아니라,  
사용자가 선택한 각 질문에 대해 개별적으로 생성되는 **확장 리포트 단위**로 구성된다.

각 질문은 독립적인 해석 단위를 가지며,  
유저는 궁금한 질문만 선택적으로 열람할 수 있다.

유료 리포트 공통 구조:

1. 현재 상태 재정리
2. 숨겨진 구조 또는 원인 제시
3. 행동 / 선택 메커니즘 설명
4. 현재 상태 재정의
5. 다음 판단 포인트 제시

---

### 3.7 사용자 타입 추론 로직 (User Type Inference Layer)

#### 3.7.1 목적

- 리포트 톤 개인화
- 질문 추천 개인화
- 유료 확장 질문 최적화

#### 3.7.2 공통 추론 축

| 축                       | 설명                                                     |
| ------------------------ | -------------------------------------------------------- |
| 인지형 vs 감정형         | 구조/이유/패턴 중심 vs 감정/관계/체감 중심               |
| 불안형 vs 안정형         | 미래/후회/타이밍에 민감 vs 현재 상태 유지 중심           |
| 회피형 vs 직면형         | 문제를 미루거나 피함 vs 불편함을 바로 봄                 |
| 자기이해형 vs 해결형     | "나는 어떤 사람인가"에 집중 vs "어떻게 해야 하나"에 집중 |
| 타인중심형 vs 자기중심형 | 타인 시선/관계 구조 중심 vs 자신의 기준/상태 중심        |

#### 3.7.3 점수 기반 추론 방식

- 각 질문 선택지에는 복수의 성향 가중치가 연결된다
- 유저의 선택에 따라 각 축의 점수가 누적된다
- 최종적으로 상위 2~3개 성향을 기반으로 개인화 적용한다

#### 3.7.4 적용 영역

| 적용 대상        | 인지형         | 감정형         | 불안형         | 회피형          |
| ---------------- | -------------- | -------------- | -------------- | --------------- |
| 무료 리포트 톤   | 구조/원리 중심 | 체감/감정 중심 | —              | —               |
| 질문 추천        | 구조 질문 우선 | 감정 질문 우선 | 미래 흐름 질문 | 자기합리화 질문 |
| 유료 리포트 방향 | 구조 뒤집기    | 감정 폭로      | 미래 흐름      | 자기합리화 깨기 |

---

### 3.8 질문 설계 원칙

- 질문 하나는 성향, 가치, 행동 패턴 중 최소 2개 이상에 기여해야 한다
- 질문은 유저의 선택 구조를 드러내야 한다 (정답 맞히기 X)
- 검사처럼 느껴지지 않으면서도, 응답 조합 시 높은 해석 밀도를 가져야 한다

## 3.9 리포트 언어 & 톤 시스템 (Report Language & Tone System)

### 3.9.1 기본 원칙

Corelog의 모든 리포트는 **일관된 반말(비격식체)** 로 작성한다.

이는 사용자에게 다음 경험을 제공하기 위함이다:

- 거리감 없는 몰입감
- “내 얘기를 정확히 짚는다”는 느낌
- 분석이 아닌 “직접 들은 해석” 같은 체감

---

### 3.9.2 톤 정의

Corelog의 톤은 다음 3가지 특성을 동시에 가져야 한다:

| 요소        | 설명                                            |
| ----------- | ----------------------------------------------- |
| 직설성      | 애매한 표현 없이 상태를 단정적으로 설명         |
| 구조성      | 감정이 아닌 패턴, 기준, 선택 구조 중심으로 설명 |
| 절제된 감정 | 과도한 공감 없이, 필요한 만큼만 감정 표현       |

---

### 3.9.2.5 분석 타입별 톤 (Self / Other / Relationship)

분석 타입에 따라 호칭과 관점이 변경된다:

| 분석 타입               | 호칭                        | 관점                     | 예시 문장                                      |
| ----------------------- | --------------------------- | ------------------------ | ---------------------------------------------- |
| **self** (나)           | "너", "넌"                  | 대상자의 내부 상태, 선택 | "너는 이런 상황에서 결정을 미루는 경향이 있어" |
| **other** (상대)        | "그 사람", "그"             | 상대방의 행동과 패턴     | "그 사람은 확신이 없을 때 회피하는 쪽이야"     |
| **relationship** (관계) | "둘 사이", "너와 그 사람은" | 두 사람 간의 반복 구조   | "둘 사이에서 반복되는 패턴은..."               |

---

### 3.9.3 문장 스타일 규칙

#### 1) 단정형 문장 사용

- “~인 것 같아” ❌
- “~일 수도 있어” ❌

→ 반드시 단정형으로 표현

- “이 상태야” ⭕
- “이 구조로 움직이고 있어” ⭕

---

#### 2) 분석 중심 문장

- 감정 묘사보다 **행동/선택 구조 설명 우선**

예:

- “불안해하고 있어” ❌
- “확신이 없을 때 결정을 미루는 쪽이야” ⭕

---

#### 3) 불필요한 완충어 제거

- “아마”, “조금”, “약간” 등 제거

→ 해석 밀도 유지

---

#### 4) 짧고 끊기는 문장 구조

- 한 문장은 1개의 의미만 전달
- 긴 문장 지양, 리듬감 있는 단문 구조 사용

---

### 3.9.4 금지 표현

다음 유형의 문장은 사용하지 않는다:

#### 1) 과도한 공감 표현

- “많이 힘들었지?”
- “너무 속상했겠다”

→ 감정 소비형 문장 금지

---

#### 2) 조언/해결 제시

- “이렇게 해보는 게 좋아”
- “이 방향이 맞아”

→ 리포트는 해석까지만 제공

---

#### 3) 점술/예언 톤

- “곧 좋은 일이 생길 거야”
- “이 관계는 끝나게 돼 있어”

→ 미래 단정 금지

---

#### 4) MBTI/성격 일반화

- “넌 원래 이런 사람이야”
- “이런 성향이 강한 타입이야”

→ 일반화 금지, 구조 기반 설명만 허용

---

### 3.9.5 리포트 흐름 내 톤 규칙

리포트는 다음 흐름을 따른다:

1. 상황 정의 → 직설적 반말
2. 패턴 설명 → 구조 중심 반말
3. 원리 설명 → 분석 중심 반말
4. 흐름 제시 → 약간의 긴장감 유지
5. 결핍 생성 → 다음 질문으로 연결

→ 전 구간 동일한 반말 유지  
→ 톤 변화는 말투가 아닌 “해석 강도”로 조절  
→ 추상적인 표현 지양

---

### 3.9.6 예시 (Good vs Bad)

#### Good

> 확신은 없는데, 그렇다고 관계를 끊지도 못하고 계속 보고 있는 상태야.  
> 결정을 해야 하는 순간을 계속 미루고 있어.

#### Bad

> 현재 확신이 부족한 상태에서 관계를 유지하고 계신 것으로 보입니다.  
> 조금 더 신중하게 생각해보시는 것이 좋을 것 같습니다.

---

### 3.9.7 UI 텍스트 톤 규칙 (UI Copy Tone)

**리포트 콘텐츠뿐만 아니라 UI 텍스트(버튼, 라벨, 안내 메시지, 에러 메시지)도 동일한 반말 규칙을 적용한다.**

#### 적용 범위

| 요소         | 규칙                                                          |
| ------------ | ------------------------------------------------------------- |
| 버튼 텍스트  | 반말, 행동형 동사 사용 (예: "확인해", "계속해")               |
| 라벨/헤더    | 반말, 직설적 표현 (예: "더 깊이 알고 싶어?", "마지막 단계야") |
| 안내 문구    | 반말, 친근한 톤 (예: "2개 이상 선택하면 자동 할인이 적용돼")  |
| 에러 메시지  | 반말, 명확한 지시 (예: "올바른 휴대폰번호를 입력해줘")        |
| 플레이스홀더 | 반말 또는 예시로 표시 (예: "010-1111-1111")                   |

#### Good vs Bad 예시

| 요소   | Bad ❌              | Good ⭕         |
| ------ | ------------------- | --------------- |
| 버튼   | "확인하기"          | "확인해"        |
| 안내   | "선택해주세요"      | "선택해봐"      |
| 에러   | "입력해주세요"      | "입력해줘"      |
| 메시지 | "완료되었습니다"    | "완료됐어"      |
| 헤더   | "마지막 단계입니다" | "마지막 단계야" |

#### 핵심 원칙

1. **존댓말 금지**: 모든 존댓말(-세요, -습니다, -ㅂ니다)은 반말로 변환
2. **거리감 제거**: 사용자와의 심리적 거리를 좁히기 위해 친근한 톤 유지
3. **일관성**: 전체 UI에서 동일한 반말 톤 유지 (리포트 콘텐츠와 통일)

### 3.10 리포트 생성 전략 레이어 (Report Strategy Layer)

#### 3.10.1 목적

사용자 응답 기반으로 추론된 **사용자 타입(User Type)** 을 활용하여  
리포트 생성 방식, 해석 방향, 질문 추천 전략을 동적으로 결정한다.

---

#### 3.10.2 핵심 구조

리포트 생성은 다음 흐름을 따른다:

1. 사용자 입력 데이터 수집
2. 사용자 타입 추론 (3.7 참조)
3. 리포트 패턴 선택
4. 패턴 기반 리포트 생성
5. 패턴 기반 질문 세트 생성

---

#### 3.10.3 리포트 패턴 정의

Corelog는 무료 리포트 생성을 위해 다음 3가지 패턴을 사용한다:

| 패턴명                            | 설명                                            |
| --------------------------------- | ----------------------------------------------- |
| Default Detect (들켰다 패턴)      | 현재 상태와 반복 구조를 정확히 짚고 결핍을 생성 |
| Uncomfortable Truth (불편한 진실) | 사용자가 인식하지 못한 불편한 구조를 드러냄     |
| Belief Break (착각 깨기)          | 사용자의 기존 해석을 뒤집고 새로운 구조를 제시  |

---

#### 3.10.4 패턴 선택 로직

패턴은 사용자 타입 및 입력 데이터를 기반으로 선택된다.

| 조건                           | 패턴                |
| ------------------------------ | ------------------- |
| 기본 상태 / 일반 유저          | Default Detect      |
| 감정 강도 높음 / 연애 카테고리 | Uncomfortable Truth |
| 자기이해형 / 사고 중심         | Belief Break        |

※ 초기 버전에서는 단순 규칙 기반으로 시작하며, 이후 데이터 기반 최적화 가능

---

#### 3.10.5 패턴별 리포트 구조

각 패턴은 고정된 구조를 가진다:

**1) Default Detect**

- 상황 정의
- 반복 패턴
- 선택 구조 설명
- 현재 흐름
- 결핍 생성

**2) Uncomfortable Truth**

- 현재 인식 인정
- 반전 해석
- 구조 설명
- 관계/상황 영향
- 추가 해석 필요성 제시

**3) Belief Break**

- 사용자 인식 제시
- 인식 반박
- 실제 원인 설명
- 선택 구조 연결
- 새로운 질문 유도

---

#### 3.10.6 질문 세트 매칭

리포트 패턴에 따라 질문 세트는 다르게 구성된다.

| 패턴                | 질문 방향                           |
| ------------------- | ----------------------------------- |
| Default Detect      | 후회 / 상대 시선 / 자기 패턴        |
| Uncomfortable Truth | 주도권 / 현실 분석 / 숨은 의도      |
| Belief Break        | 오해된 기준 / 선택 구조 / 판단 오류 |

---

#### 3.10.7 결핍 문장 규칙

모든 무료 리포트는 마지막에 반드시 다음 구조를 포함해야 한다:

- 현재 해석이 완전하지 않다는 암시
- 더 중요한 질문이 남아 있다는 인식
- 추가 탐색 필요성 강조

예:

- "근데 여기서 하나 빠져 있어"
- "이건 아직 안 나왔어"
- "이걸 안 보면 같은 선택이 반복될 가능성이 높아"

---

#### 3.10.8 확장성

- 초기 버전: 규칙 기반 패턴 선택
- 확장: 전환율 기반 패턴 최적화 (A/B 테스트)
- 장기: 사용자 세그먼트별 패턴 자동 최적화

### 5.3 인증 & 세션 구조 (Auth & Session)

#### 회원 인증

- **방식**: Supabase Auth (소셜 로그인만 지원)
  - 지원: Google, Kakao, Apple (추후 확장 가능)
  - 플로우: 소셜 로그인 → Supabase에서 회원 자동 생성 → 세션 생성
- **보호 라우트**: `/mypage/*` — 미인증 시 `/login` 리다이렉트 (Next.js Middleware)
- **관리자 라우트**: `/admin/*` — 별도 관리자 Role 검증, 미인증 시 `/admin/login` 리다이렉트 (관리자는 이메일/비밀번호 로그인)

#### 비회원 세션

- **리포트 접근**: `session-id` (UUID) + 전화번호 뒤 4자리 비밀번호 조합으로 조회
  - 비밀번호는 결제 시 사용자가 직접 설정 (4자리 숫자)
  - 예: 010-1234-5678 → 비밀번호는 5678 아님, 사용자가 입력한 다른 4자리
  - **보안 처리**: 비밀번호는 bcrypt로 해시화하여 DB 저장 (평문 저장 금지)
  - **조회 검증**: 입력한 비밀번호를 해시화 후 DB 저장된 해시와 비교 검증
- **session-id**: URL에 포함 (`/report/[session-id]`), 서버에서 소유자 검증
- **세션 만료**: 비회원 리포트는 결제 완료 후 **180일** 보관
- **만료 시**: 만료 안내 페이지로 이동 (`"이 리포트는 보관 기간이 지났어요"`)

#### 비회원 기록 조회 (`/guest/lookup`)

**UI 구조**

- 타이틀: `"리포트 조회"`
- 본문: `"전화번호랑 비밀번호로 찾을 수 있어."`
- 입력 필드:
  - 전화번호 (라벨: `"전화번호"`, 플레이스홀더: `"010-0000-0000"`)
  - 비밀번호 (라벨: `"비밀번호"`, 플레이스홀더: `"숫자 4자리"`)
- 제출 버튼: `"찾아보기"`
- 하단 설명: `"비밀번호는 결제할 때 설정한 4자리 숫자야"`

**플로우**

- 입력: 전화번호 + 비밀번호 (결제 시 사용자가 설정한 4자리 숫자)
- 제출:
  1. 전화번호로 비회원 주문 검색
  2. 비밀번호 대조
  3. 일치 시: 해당 session-id의 리포트로 이동 (`/report/[session-id]`)
  4. 불일치 시: `"일치하는 기록을 찾을 수 없어요."` 안내
  5. 3회 연속 실패 시: `"3회 실패했어요. 1분 후 다시 시도해주세요."` 후 1분 잠금
- **비밀번호 규칙**: 4자리 숫자만 허용
- **개인정보 수집 동의**: 결제 전 전화번호 수집 명시 및 PIPA 기반 개인정보 이용약관 동의 받음

#### 회원 전환 유도

- 비회원 결제 완료 후 리포트 페이지 상단에 CTA 배너 노출
- 문구: `"기록을 영구 보관하고 싶다면 회원가입 →"`
- 가입 완료 시 해당 세션과 회원 계정 자동 연결

### 5.4 카테고리 진입 구조

카테고리는 단순 주제 분류가 아니라 이후 질문 구조를 결정하는 1차 진입점이다.

- 연애와 인간관계는 분리 (썸은 연애 카테고리에 포함)
- 각 카테고리는 서로 다른 해석 축과 질문 후보를 가진다

---

### 5.5 질문 UI 구조 (Question Interface)

#### 기본 원칙

- 모든 질문은 선택지 기반으로 제공한다
- 질문 성격에 따라 단일 선택 또는 복수 선택으로 설계한다
- 초기 버전에서는 자유 입력 없이, 시스템이 생성한 선택지 기반 질문만 제공한다

#### 질문 노출 방식

- 현재 질문에 적절한 선택지 4~6개 노출
- 진행 상태 표시(프로그레스 바)는 최소화
- 질문 사이 전환은 부드럽고 직관적이어야 한다

#### 질문 단계 원칙

1. 현재 상황 정의
2. 현재 감정 확인
3. 선택 시 중요 기준 확인
4. 불확실할 때의 반응 패턴 확인
5. 반복되는 구조 확인
6. 자기 인식 보정

---

### 5.6 /analyze 상태 관리

#### 클라이언트 상태

- 질문 응답은 **클라이언트 전역 상태** (Zustand 또는 Context API)로 관리
- 각 단계 이동 시 서버에 중간 저장하지 않음 → 최종 제출 시 한 번에 전송

#### 뒤로가기 / 수정

- 이전 질문으로 복귀 허용
- 이전 답변 수정 가능 (수정 시 이후 단계 초기화)

#### 새로고침 복구

- 각 단계 이동 시 `sessionStorage`에 임시 저장
- 새로고침 시 복구 시도
- 복구 실패 시: `"이전 응답이 사라졌어요. 처음부터 시작할게요"` 안내 후 초기화

#### 이탈 방지 UX

- 분석 플로우 진행 중 페이지 이탈 시도 (브라우저 뒤로가기, 창 닫기 등):
  - 모달/다이얼로그: `"작성 중인 응답이 사라질 수 있어요. 정말 나갈까요?"`
  - 버튼: `"계속 작성" / "나가기"`
  - 나가기 시 `sessionStorage`에 저장된 임시 데이터 제거
- 헤더의 로고 클릭 또는 특정 링크 클릭 시에도 동일하게 처리

#### 질문 분기 로직

- 서버(API Route)에서 이전 응답을 기반으로 다음 질문을 반환
- 클라이언트는 서버 응답을 받아 질문 렌더링만 담당

### 5.8 에러 & 엣지케이스 처리

| 상황                          | 처리 방식                                                                                |
| ----------------------------- | ---------------------------------------------------------------------------------------- |
| AI 리포트 생성 실패           | `"분석에 문제가 생겼어요. 잠시 후 다시 시도해주세요"` + 재시도 버튼                      |
| 결제 완료 후 리포트 생성 실패 | 관리자 텔레그램 알림 자동 발송 + `"결제는 완료됐어요. 잠시 후 리포트가 생성됩니다"` 안내 |
| 잘못된 session-id 접근        | 404 페이지: `"해당 리포트를 찾을 수 없어요"`                                             |
| 세션 만료 (비회원 180일 초과) | 만료 안내 페이지                                                                         |
| 네트워크 오류                 | 각 API 호출에 retry 1회, 실패 시 하단 토스트 에러 메시지                                 |
| 이미 구매한 질문 재선택       | 체크박스 비활성화 + `"이미 구매한 질문이에요"` 툴팁                                      |
| 비회원 조회 실패              | `"일치하는 기록을 찾을 수 없어요"`                                                       |
| 관리자 미인증 접근            | `/admin/login` 리다이렉트                                                                |
| AI 생성 15초 초과             | 타임아웃 에러 + 재시도 버튼 + 관리자 텔레그램 알림                                       |

#### 에러 페이지 정의

**404 페이지 (`not-found.tsx`)**

- 메시지: `"해당 페이지를 찾을 수 없어요"`
- 부제: `"존재하지 않는 리포트이거나 링크가 잘못되었을 수 있습니다"`
- CTA: `"메인으로 돌아가기"` → `/` 이동
- 디자인: 브랜드 색상 유지, 불안감 없이 따뜻한 톤

**500 페이지 (`error.tsx`)**

- 메시지: `"서비스에 일시적 오류가 발생했어요"`
- 부제: `"잠시 후 다시 시도해주세요"`
- CTA: `"다시 시도" / "메인으로"`
- 자동 재시도: 5초 후 자동 새로고침 시도 (선택사항)
- 관리자 알림: 500 에러 발생 시 관리자 텔레그램 알림 발송

### 6-2. 분석 플로우 (`/analyze`)

분석 타입(`type=self|other|relationship`)은 랜딩에서 선택 후 진입

#### Step 0. 카테고리 선택

분석 타입별 카테고리 질문:

**type=self (나)**

- 질문: `"나는 어떤 부분이 궁금해?"`
- 선택지: 연애 / 감정 / 인간관계 / 직업/진로

**type=other (상대)**

- 질문: `"저 사람은 어떤 부분이 궁금해?"`
- 선택지: 연애 / 감정 / 인간관계 / 직업/진로

단일 선택만 가능

#### Step 1. 하위 분기 선택 (카테고리별)

**연애 카테고리**: 썸 / 연애 중 / 이별 / 재회 중 선택  
**인간관계 카테고리**: 친구 / 가족 / 직장 / 사람 전반 중 선택  
**직업/진로 카테고리**: 이직 / 방향성 / 확신 부족 / 현실 vs 적성 중 선택  
**감정 카테고리**: 하위 분기 없음 (Step 2로 진입)

단일 선택만 가능
이후 질문 구조와 리포트 톤이 이 선택에 따라 결정됨

#### Step 2. 상황 정의

- 질문: `"지금 어떤 상황이야?"`
- 선택지 기반, 복수 선택 가능
- 초기 버전에서는 자유 입력 없이 시스템이 생성한 선택지만 제공

#### Step 3. 감정 / 기준 / 행동 질문

- 감정형 질문 (현재 체감 상태)
- 기준형 질문 (선택 시 중요 요소)
- 행동형 질문 (불확실할 때 반응)
- 반복 패턴 질문

#### Step 4. 분석 세션 생성

- 모든 응답을 서버에 한 번에 전송
- `/report/[session-id]` 이동 (분석 타입, 카테고리, 응답 데이터 포함)
- 로딩: 문구 순환 (5.8 참조)

### 6-3. 리포트 페이지 (`/report/[session-id]`)

무료 리포트와 유료 리포트를 분리된 페이지로 제공하지 않는다.
하나의 단일 리포트 페이지 안에서 확장되는 구조다.

#### 6-3.1 페이지 구조

1. **무료 기본 리포트**
2. **질문 선택 섹션**
3. **유료 확장 리포트 영역** (결제 후 점진 추가)

#### 6-3.2 무료 기본 리포트

- 유저 상태 묘사 → 성향/패턴 → 반복 구조 → 작동 원리 → 흐름 제시 → 핵심 깨달음 한 문장
- `"이거 내 얘기네"` + `"들킨 느낌"` 제공
- 모든 해석을 닫지 않고, 다음 질문이 생기게 만든다

#### 6-3.3 질문 선택 섹션

- 질문은 현재 리포트 내용 + 사용자 타입 기반으로 동적 생성
- 추천 질문 3개를 우선 노출하고, 나머지 질문은 아코디언 영역에서 탐색할 수 있게 한다
- 각 탐색 단계에서 노출되는 질문 세트는 초기 버전 기준으로 추천 질문 3개 + 추가 질문 5개, 총 8개로 고정한다
- Full Question Bundle은 해당 단계에서 노출된 전체 8개 질문 세트를 기준으로 한다
- 초기 버전에서는 자유 입력 없이, 시스템이 생성한 질문 세트를 기준으로 탐색을 진행한다

#### 6-3.4 질문 생성 원칙

- 사용자 타입 기반 개인화
- 서로 다른 사고 축으로 구성 (의미상 중복 질문 금지)
- `"설명"` 보다 `"다음 탐색 욕구"` 자극

#### 6-3.5 질문 카드 노출 구조

질문은 단순 제목 리스트가 아니라,  
**질문 제목 + 짧은 설명 1줄** 구조의 카드 형태로 노출한다.

##### 질문 카드 구성

- 질문 제목
- 이 질문을 왜 봐야 하는지에 대한 짧은 설명
- 선택 상태 UI (체크박스 또는 선택 버튼)

##### 예시

- **지금 내가 놓치고 있는 판단 기준은 뭐야?**  
  답답한데도 정리가 안 되는 이유가 기준 때문인지 보여줘.

- **나는 왜 항상 실행 직전에서 멈추게 될까?**  
  생각은 하는데 행동이 안 이어지는 이유를 짚어줘.

##### 설계 원칙

- 질문 제목만으로 끝내지 않는다
- 설명은 1문장 이내로 짧게 유지한다
- 설명은 “정보 소개”보다 “탐색 욕구 자극”에 초점을 둔다

---

#### 6-3.6 CTA 버튼

- 사용자가 질문을 2개 이상 선택했거나 아코디언 질문까지 탐색한 경우,
- 현재 단계 전체 질문 세트를 한 번에 열람할 수 있는
- Full Question Bundle CTA를 함께 노출할 수 있다
- Full Question Bundle CTA 예시: `현재 단계 전체 보기 ₩4,900`
- Sticky 하단 영역에 노출
- 버튼 문구는 "결제"보다 "탐색" 중심으로 작성

CTA 상단 안내 문구:

- `"지금 이 상태를 더 정확하게 이해하고 싶다면"`
- `"여기서 멈추면 아직 보이지 않는 부분이 있어"`
- `"다음 질문을 보면 흐름이 더 명확해져"`

##### CTA 상태 변화 규칙

사용자의 질문 선택 상태에 따라 CTA는 동적으로 변경된다.

| 상태                | CTA 문구                   | 버튼 상태       |
| ------------------- | -------------------------- | --------------- |
| 선택 없음           | 질문을 선택해봐            | 비활성화 (회색) |
| 1개 선택            | 이 질문의 답 보기 ₩900     | 활성화          |
| 2개 선택            | 선택한 질문 보기 ₩1,600    | 활성화          |
| 3개 선택            | 선택한 질문 보기 ₩2,100    | 활성화          |
| 현재 단계 전체 보기 | 현재 단계 전체 보기 ₩4,900 | 활성화          |

##### 추가 표시 요소

- 2개 이상 선택 시 절약 금액 함께 표기
- 3개 선택 시 “가장 많이 선택됨” 또는 “추천 조합” 뱃지 노출 가능

---

#### 6-3.7 리포트 렌더링 구조

| 상황              | 처리 방식                                                    |
| ----------------- | ------------------------------------------------------------ |
| 최초 진입         | 무료 리포트만 렌더링 (Server Component, SSR)                 |
| 유료 결제 완료 후 | 해당 섹션 클라이언트에서 추가 렌더링 (CSR, AI 스트리밍)      |
| 새로고침          | 이미 결제된 질문 목록 서버에서 조회 → 전체 리포트 복원 (SSR) |

- **유료 리포트 노출 방식**: AI 스트리밍 (`ReadableStream`) + 타이핑 효과
- **결제 후 UX**: 모달 닫힘 → 해당 섹션으로 자동 스크롤 → 스트리밍 시작

### 6-4. 결제 처리 구조 (Payment Layer)

- 별도의 결제 페이지를 사용하지 않는다. 리포트 페이지 내에서 처리된다.
- 선택한 질문 또는 현재 단계 전체 질문 세트에 대한 유료 리포트가 즉시 하단에 추가 (스트리밍)

#### UI 형태

| 디바이스 | UI        |
| -------- | --------- |
| 모바일   | 바텀시트  |
| 데스크탑 | 센터 모달 |

Toss Payments 위젯은 바텀시트/모달 안에서 렌더링된다.

---

#### 결제 흐름

질문 선택  
→ 바텀시트/모달 노출 (Toss Payments 위젯)  
→ 결제 진행  
→ Toss 콜백 → /payment/success?orderId=...&paymentKey=...  
→ 서버 검증 (금액, 상태 확인)  
→ 리포트 생성 시작  
→ /report/[session-id] 복귀 + 해당 섹션 스크롤

---

#### 결제 UI 상세

- 카드 / 간편결제 지원
- **회원**: 로그인 상태로 자동 결제 (이름, 전화번호 Supabase에서 자동 조회)
- **비회원**: 전화번호 + 비밀번호(4자리) 입력 필수
  - 비밀번호는 향후 기록 조회 시 사용됨 (`/guest/lookup`)
- 결제 중 로딩: 버튼 비활성화 + 스피너
- 결제 실패 시: 모달 내 에러 메시지 + 재시도 버튼 (모달 유지, 페이지 이동 X)
- `/payment/fail` 콜백: 실패 사유 확인 후 리포트 페이지로 복귀 + 실패 토스트 노출

---

#### 결제 후 동작

- 선택한 질문 또는 현재 단계 전체 질문 세트에 대한 유료 리포트가 즉시 하단에 추가 (스트리밍)
- 페이지 이동 없음

### 6-5. 확장 리포트 구조 (Expanded Reports)

유료 결제 완료 후 선택한 질문에 대한 유료 리포트가 **동일 페이지 하단에 추가**된다.

#### 확장 방식

- 최초 진입: 무료 기본 리포트만 노출
- 질문 선택 또는 Full Question Bundle 결제 후: 현재 단계 유료 리포트 추가
- 다음 단계 질문 선택 또는 Full Question Bundle 결제 후: 다음 단계 유료 리포트가 그 아래에 추가
- 탐색 루프 최대 **3회**까지 허용

---

#### 유료 리포트 구조

각 질문은 하나의 독립 리포트로 제공:

1. 기존 상태 재정리
2. 숨겨진 구조 또는 원인 제시
3. 행동 / 선택 메커니즘 설명
4. 현재 상태 재정의
5. 다음 판단 포인트 제시

---

#### 리포트 UX

- 질문별 카드 / 아코디언 구조
- 단계적으로 열리는 방식 (한 번에 모두 펼치지 않음)
- 무료 리포트와 유료 리포트는 톤은 이어지되, 깊이감은 분명히 달라야 한다

#### 핵심 경험 흐름

"아 이게 나네" → 무료 리포트  
"헉, 이건 생각 못했는데" → 유료 확장 리포트  
"그래서 이제 어떻게 봐야 하지?" → 다음 질문 선택

---

#### 추가 질문 추천 및 재구성 원칙 (루프 핵심)

각 유료 리포트 하단에는  
아직 선택하지 않은 다음 질문 2~3개를 개인화하여 추천한다.

이때 질문은 기존 리스트를 그대로 반복 노출하지 않고,  
**현재까지 열린 리포트 내용 + 사용자 타입 + 이미 선택한 질문 이력**을 반영하여  
다음 질문 세트를 새로 재구성한다.

---

##### 추천 원칙

- 현재까지 읽은 리포트 내용 기반
- 사용자 타입 기반 개인화
- 이미 선택하여 열람한 질문은 다시 추천하지 않음
- 동일 질문 축 또는 의미가 겹치는 질문은 제거 또는 교체
- 한 단계 더 깊고, 더 구체적인 질문으로 이어져야 함

---

##### 질문 진화 방향

- 무료 리포트 단계: 상태 이해 중심
- 유료 리포트 1단계 이후: 원인 / 구조 이해 중심
- 유료 리포트 2단계 이후: 선택 / 행동 / 우선순위 이해 중심

---

##### 추천 방향 예시

| 이미 본 질문 | 다음 추천 방향       |
| ------------ | -------------------- |
| 원인 질문    | 선택 질문            |
| 선택 질문    | 우선순위 / 행동 질문 |
| 미래 질문    | 기준 / 구조 질문     |

---

##### 단계별 예시 흐름

| 단계        | 질문 방향                                               |
| ----------- | ------------------------------------------------------- |
| 무료 이후   | 왜 이런 상태인지                                        |
| 유료 1 이후 | 왜 이 구조가 반복되는지 / 무엇을 놓치고 있는지          |
| 유료 2 이후 | 실제로 어떤 선택을 하게 되는지 / 무엇부터 바꿔야 하는지 |

---

##### UI 처리 원칙

- 이미 구매한 질문은 비활성화하거나 숨김 처리 가능
- 새 질문 세트는 기존 질문 영역을 대체하거나,  
  `"여기서 더 보면 좋은 질문"` 영역으로 재노출한다
- 추천 질문은 항상 2~3개만 우선 노출하고,  
  필요 시 추가 질문은 아코디언 영역에서 탐색 가능하게 한다

#### 탐색 루프 깊이 단계

| 단계  | 내용                        |
| ----- | --------------------------- |
| 1단계 | 상태 이해                   |
| 2단계 | 구조 / 원인 이해            |
| 3단계 | 선택 / 행동 / 우선순위 이해 |

#### 루프 3회 이후 처리

- **최대 탐색**: 무료 리포트 → 유료 1단계 → 유료 2단계 → 유료 3단계 (최대 3회 루프)
- **3단계 완료 후**: 질문 선택 영역 숨김 처리
- **후행 상품 제시**:
  - 구독 모델 안내: `"월간 변화 추적으로 패턴을 더 깊게 알아봐"`
  - 다른 카테고리 제안: `"다른 분야도 함께 분석해보면 더 명확해져"`

### 9.6 질문 번들 선택 유도 전략 (Bundle Selection Strategy)

#### 9.6.1 목적

질문 번들 구조는 단순 할인 제공이 아니라,
사용자가 **복수 질문을 자연스럽게 선택하도록 유도하여 평균 객단가를 상승시키는 것**을 목표로 한다.

---

#### 9.6.2 가격 및 번들 구조

질문은 개별 선택과 번들 선택이 모두 가능하며,
복수 선택 시 할인 구조를 적용한다.

| 선택 개수 | 가격    | 설명          |
| --------- | ------- | ------------- |
| 1개       | 900원   | 기본 가격     |
| 2개       | 1,600원 | 약 200원 절약 |
| 3개       | 2,100원 | 약 600원 절약 |

##### 설계 원칙

- 선택 개수가 증가할수록 단가가 낮아지는 구조
- 3개 선택 시 할인 체감이 명확하게 느껴지도록 설계
- 가격은 A/B 테스트를 통해 조정 가능

---

#### 9.6.3 절약 금액 시각화

사용자가 번들 선택 시 할인 체감을 직관적으로 인지할 수 있도록
절약 금액을 UI에 명확히 표시한다.

##### UI 규칙

- 선택 개수에 따라 실시간으로 절약 금액 표시
- 가격 인접 영역 또는 버튼 하단에 강조 텍스트로 노출

##### 예시

- 2개 선택 시 → `200원 절약`
- 3개 선택 시 → `600원 절약 🔥`

##### 설계 원칙

- 절약 금액은 반드시 숫자로 명시
- 가격보다 “이득을 보고 있다”는 인식 유도

---

#### 9.6.4 베스트 선택 유도 (Best Choice Highlighting)

특정 선택 개수(주로 3개)에 대해
사용자가 자연스럽게 선택하도록 강조 UI를 제공한다.

##### UI 요소

- “가장 많이 선택됨”
- “추천 조합”
- “이 조합으로 보면 더 명확해짐”

##### 적용 위치

- 3개 선택 시 가격 영역 또는 CTA 상단
- 시각적으로 구분되는 뱃지/텍스트 사용

##### 설계 목적

- 사회적 증거(Social Proof) 제공
- 선택 고민을 줄이고 자연스럽게 다중 선택 유도

---

#### 9.6.5 질문 구성 전략 (Cross-Axis Question Composition)

번들 구매 유도를 위해 질문은 반드시
**서로 다른 사고 축을 기반으로 구성**되어야 한다.

##### 핵심 원칙

- 동일 의미 또는 유사 질문 중복 금지
- 각 질문은 서로 다른 관점/욕구를 자극해야 함

##### 권장 질문 축

| 축               | 설명                          |
| ---------------- | ----------------------------- |
| 미래 (후회)      | 선택 결과 및 후회 가능성      |
| 타인 (관계/시선) | 상대가 나를 어떻게 인식하는지 |
| 자기 (패턴)      | 반복되는 선택 및 행동 구조    |

##### 예시 조합

- “이 선택, 나중에 후회할 가능성은?”
- “상대는 지금 나를 어떻게 보고 있을까?”
- “나는 왜 항상 이런 선택을 반복할까?”

##### 설계 효과

- 하나만 선택하기 어려운 구조 형성
- 자연스럽게 2개 → 3개 선택으로 확장
- “이왕이면 같이 봐야 한다”는 인식 형성

---

#### 9.6.6 번들 맥락 내 카피 적용 원칙 (Bundle Copy Context)

번들 영역에서는 일반 CTA 카피 원칙(8.5)을 그대로 사용하되,
“가격”이 아닌 “해석 완성” 중심으로 강조한다.

##### 적용 방식

- 단순 할인 문구 대신, 질문 조합의 의미를 강조
- 선택 개수가 늘어날수록 “더 완전한 해석”이라는 인식 제공

##### 권장 표현 예시

- “이 질문까지 보면 흐름이 더 명확해짐”
- “여기서 멈추면 중요한 부분이 빠짐”
- “이 조합으로 보면 전체 구조가 보임”

##### 금지 방향

- 할인 자체만 강조하는 표현
- 가격 절약만을 중심으로 한 설득

##### 핵심 원칙

- 번들은 가격이 아니라 “해석의 완성도”로 설득한다

---

#### 9.6.7 핵심 설계 요약

- 번들은 가격 전략이 아니라 행동 유도 구조
- 할인보다 “선택 필요성” 인식이 중요
- 질문은 반드시 서로 다른 축으로 구성
- UI는 절약 + 추천을 동시에 전달해야 함

### 9.7 탐색 루프 구조 (Exploration Loop)

```
무료 리포트
→ 질문 선택 또는 Full Question Bundle 결제 (1단계)
→ 유료 확장 리포트
→ 추가 질문 생성
→ 질문 선택 또는 Full Question Bundle 결제 (2단계)
→ 유료 확장 리포트
→ 추가 질문 생성
→ 질문 선택 또는 Full Question Bundle 결제 (3단계)
→ 종료
```

- 탐색 루프 **최대 3회** 허용
- 동일 질문 축 반복 금지
- 루프 진행할수록 더 깊은 질문으로 이동해야 한다

#### 11.2.3 profiles (회원 프로필)

> `auth.users` 와 1:1 연결되는 확장 프로필. 소셜 로그인 완료 시 트리거에 의해 자동 생성.

| 컬럼              | 타입          | NULL 여부 | 기본값   | 설명                                           |
| ----------------- | ------------- | :-------: | -------- | ---------------------------------------------- |
| `id`              | `uuid`        | NOT NULL  | —        | PK, FK → `auth.users.id` ON DELETE CASCADE     |
| `nickname`        | `text`        | NOT NULL  | —        | 소셜 로그인 표시명으로 초기 세팅               |
| `email`           | `text`        | NOT NULL  | —        | `auth.users.email` 복사 (읽기 편의용 역정규화) |
| `phone`           | `text`        |   NULL    | —        | 선택 입력 전화번호                             |
| `social_provider` | `text`        | NOT NULL  | —        | `google` \| `kakao` \| `apple`                 |
| `role`            | `text`        | NOT NULL  | `'user'` | `user` \| `admin`                              |
| `created_at`      | `timestamptz` | NOT NULL  | `now()`  | 가입 시각                                      |
| `updated_at`      | `timestamptz` | NOT NULL  | `now()`  | 마지막 수정 시각 (트리거 자동 갱신)            |

**인덱스**: `email` (UNIQUE), `role`

#### 11.2.4 analysis_sessions (분석 세션)

> 회원/비회원 통합 핵심 테이블. `/report/[id]` URL에 노출되는 id를 보유.  
> 비회원은 `phone_encrypted` + `password_hash`로 인증하고, 회원은 `user_id`로 연결.

| 컬럼                 | 타입          | NULL 여부 | 기본값              | 설명                                                                        |
| -------------------- | ------------- | :-------: | ------------------- | --------------------------------------------------------------------------- |
| `id`                 | `uuid`        | NOT NULL  | `gen_random_uuid()` | PK, URL 노출용 session-id                                                   |
| `user_id`            | `uuid`        |   NULL    | —                   | FK → `profiles.id` ON DELETE SET NULL (회원 연결, 비회원 NULL)              |
| `phone_encrypted`    | `text`        |   NULL    | —                   | 비회원 전화번호 AES-256 암호화 (회원이면 NULL)                              |
| `password_hash`      | `text`        |   NULL    | —                   | 비회원 비밀번호 4자리 bcrypt 해시 (회원이면 NULL)                           |
| `category`           | `text`        | NOT NULL  | —                   | `love` \| `relationship` \| `career` \| `emotion`                           |

#### 11.2.12 guest_lookup_attempts (비회원 조회 시도)

> 비회원 기록 조회 `/guest/lookup` 의 실패 횟수 추적. 3회 실패 시 1분 잠금.

| 컬럼           | 타입          | NULL 여부 | 기본값              | 설명                                |
| -------------- | ------------- | :-------: | ------------------- | ----------------------------------- |
| `id`           | `uuid`        | NOT NULL  | `gen_random_uuid()` | PK                                  |
| `phone_hash`   | `text`        | NOT NULL  | —                   | 전화번호 SHA-256 해시 (원본 비노출) |
| `attempted_at` | `timestamptz` | NOT NULL  | `now()`             | 시도 시각                           |

**인덱스**: `(phone_hash, attempted_at)` — 최근 1분 내 시도 횟수 집계

**rate limit 쿼리 패턴:**

```sql
SELECT COUNT(*) FROM guest_lookup_attempts
WHERE phone_hash = $1
  AND attempted_at > NOW() - INTERVAL '1 minute';
-- 결과 >= 3 이면 재시도 거부, retryAfterSeconds 반환
```

**TTL 정리**: `attempted_at < NOW() - INTERVAL '1 hour'` 레코드 주기 삭제 (pg_cron)

#### 11.2.14 트리거 & 함수

**1) 신규 회원 프로필 자동 생성**

```sql
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO public.profiles (id, email, nickname, social_provider)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data->>'full_name', split_part(NEW.email, '@', 1)),
    COALESCE(NEW.raw_user_meta_data->>'provider', 'unknown')
  );
  RETURN NEW;
END;
$$;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();
```

**2) `updated_at` 자동 갱신 (공용 함수)**

```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;

-- 적용 대상 테이블: profiles, analysis_sessions, orders, paid_reports
CREATE TRIGGER set_updated_at
  BEFORE UPDATE ON analysis_sessions
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
-- (나머지 3개 테이블에도 동일 트리거 생성)
```

**3) 비회원 세션 만료 시각 자동 설정**

```sql
CREATE OR REPLACE FUNCTION set_guest_session_expiry()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  -- user_id가 없으면 비회원 → 180일 만료
  IF NEW.user_id IS NULL THEN
    NEW.expires_at = now() + INTERVAL '180 days';
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER on_analysis_session_created
  BEFORE INSERT ON analysis_sessions
  FOR EACH ROW EXECUTE FUNCTION set_guest_session_expiry();
```

### 11.3 API Routes (`src/app/api/`)

#### A. 인증

| 메서드 | 경로                 | 설명                                                              | 인증 |
| ------ | -------------------- | ----------------------------------------------------------------- | ---- |
| `GET`  | `/api/auth/callback` | Supabase OAuth 콜백 처리 → 세션 쿠키 설정 후 `/mypage` 리다이렉트 | 없음 |

**Request**: Supabase가 전달하는 `code` 쿼리 파라미터  
**Response**: 307 리다이렉트 (`/mypage` 또는 `/?error=auth_failed`)

---

#### B. 스트리밍 리포트

| 메서드 | 경로                                                       | 설명                          | 인증                    |
| ------ | ---------------------------------------------------------- | ----------------------------- | ----------------------- |
| `GET`  | `/api/reports/[session-id]/free-report/stream`             | 무료 리포트 SSE 스트리밍 생성 | session 소유권 검증     |
| `GET`  | `/api/reports/[session-id]/questions/[question-id]/stream` | 유료 리포트 SSE 스트리밍 생성 | 결제 완료 + 소유권 검증 |

**`/api/reports/[session-id]/free-report/stream`**

- 응답 형식: `text/event-stream` (SSE)
- 처리 흐름: session 조회 → `free_report_status` = `generating` 업데이트 → Claude API 스트리밍 호출 → chunk 전송 → 완료 시 `free_report` 필드 저장 + `free_report_status` = `completed`
- 에러 시: `free_report_status` = `failed` + 텔레그램 알림 발송
- 15초 초과 시: 타임아웃 이벤트 전송 + 텔레그램 알림

**`/api/reports/[session-id]/questions/[question-id]/stream`**

- 사전 조건: `orders` 테이블에서 해당 `question_id` 결제 완료 확인
- 응답 형식: `text/event-stream` (SSE)
- 처리 흐름: 결제 확인 → `paid_reports.status` = `generating` 업데이트 → Claude API 스트리밍 호출 → chunk 전송 → 완료 시 `paid_reports.content` 저장 + `status` = `completed`
- 이미 생성된 리포트: DB에서 조회 후 즉시 전송 (재스트리밍 없음)

### 11.5 결제 흐름 상세 (Payment Flow Detail)

```
[클라이언트] 질문 선택 → CTA 클릭
  → [Server Action] 주문 예약: orders 레코드 생성 (status: pending)
  → [클라이언트] Toss Payments 위젯 렌더링 (orderId 전달)
  → [Toss] 사용자 결제 진행
  → [Toss] /payment/success?orderId=...&paymentKey=...&amount=... 콜백
  → [클라이언트] POST /api/payments/confirm 호출
  → [API Route] Toss 서버 검증 → orders 확정 → paid_reports 생성
  → [클라이언트] /report/[session-id] 복귀 + 해당 섹션 스크롤
  → [클라이언트] GET /api/reports/[session-id]/questions/[question-id]/stream 연결
  → [API Route] Claude AI 스트리밍 → 타이핑 효과로 렌더링
```

#### 결제 실패 흐름

```
[Toss] /payment/fail?code=...&message=... 콜백
  → [클라이언트] 실패 사유 파싱 → orders.status = failed 업데이트 (Server Action)
  → 리포트 페이지 복귀 + 실패 토스트 노출
```

### 12.5 핵심 E2E 테스트 시나리오 (Critical Test Paths)

아래 시나리오가 테스트 케이스로 작성되어 있는지 확인한다.

#### 시나리오 1 — 비회원 무료 플로우 (Happy Path)

```
랜딩 페이지 접속
→ 카테고리 선택 (예: 연애)
→ 하위 분기 선택 (예: 썸)
→ 질문 응답 순서대로 완료
→ 무료 리포트 페이지 정상 렌더링
→ 리포트 내용 노출 확인
```

- [x] 각 단계 진입 가능한가
- [x] 응답 선택 시 다음 질문으로 정상 이동하는가
- [x] 무료 리포트가 생성되고 렌더링되는가

**극단적 케이스**

- [x] 질문 응답 중 브라우저 **뒤로가기**를 눌렀을 때 이전 질문 상태가 올바르게 복원되는가 (또는 처음부터 재시작 안내가 표시되는가) — popstate 이벤트 인터셉트로 이전 단계 이동 구현
- [x] 질문 응답 중 **새로고침(F5)** 시 세션 데이터가 유지되거나, 처음부터 재시작 안내가 표시되는가 — sessionStorage 저장 + 복원 배너("이어서 하기/새로 시작") 구현
- [x] 답변을 선택하지 않고 **다음 버튼을 강제 클릭**하려 할 때 유효성 검사가 막아주는가 — `disabled={!hasAnswer}` 처리됨
- [x] **같은 탭/윈도우를 여러 개** 열고 동시에 서로 다른 카테고리 분석을 진행할 때 세션이 충돌하지 않는가 — 카테고리별 별도 sessionId 키 사용
- [x] 분석 진행 중 **다른 페이지로 이동 후 뒤로가기**로 돌아왔을 때 진행 상태가 유지되는가 — sessionStorage에 진행 상태 저장, 복귀 시 배너로 복원
- [x] 이미 완료된 세션 URL (`/report/[session-id]`)에 접근 후 **분석 시작 페이지로 뒤로가기** 시 중복 분석이 생성되지 않는가 — 더미 데이터 구조상 동일 sessionId 재사용, 중복 오더 없음
- [x] 모바일 환경에서 **빠른 탭 연타**로 카테고리/분기/답변을 연속 선택할 때 UI가 깨지거나 중복 요청이 발생하지 않는가 — `isSubmittingRef`로 중복 제출 방지

#### 시나리오 2 — 유료 리포트 구매 플로우 (Happy Path)

```
무료 리포트 페이지
→ 유료 질문 목록 확인
→ 질문 선택
→ Toss 결제 위젯 렌더링 확인
→ 테스트 카드로 결제 진행
→ /payment/success 콜백 처리
→ AI 스트리밍 리포트 타이핑 효과 확인
```

- [x] 유료 질문 목록이 8개 표시되는가
- [x] 결제 위젯이 orderId와 함께 정상 렌더링되는가
- [x] 결제 성공 후 리포트 페이지로 복귀하는가 — `successUrl`에 `sessionId` 파라미터 포함하여 `/payments/success` → `/report/[id]` 복귀
- [x] AI 응답이 스트리밍(타이핑 효과)으로 표시되는가 — 구매 완료 질문의 `report` 필드를 `TypewriterText`로 렌더링

**극단적 케이스**

- [x] **결제 위젯 로딩 중** 결제 버튼을 클릭하려 할 때 버튼이 비활성화되거나 동작하지 않는가 — `disabled={!widgetReady || isPaying}` 처리
- [x] 결제 진행 중 **브라우저 새로고침** 시 `orderId` 중복 생성 없이 안전하게 처리되는가 — `readPendingOrder()`로 모달 상태 복원, 재결제 시 새 orderId 발급
- [ ] `/payment/success?orderId=...&paymentKey=...&amount=...` URL을 **직접 조작**(amount 변조, paymentKey 위조)하여 접근 시 서버 측 검증이 차단하는가 — **백엔드 연동 시 처리**
- [x] **동일한 `/payment/success` URL을 두 번 연속으로** 접근(뒤로가기 후 재접근 등) 시 중복 결제 처리가 발생하지 않는가 — `sessionStorage['payment_processed_{orderId}']` 체크 후 리포트 페이지로 자동 리다이렉트
- [x] 이미 구매 완료된 질문을 **다시 체크박스로 선택**하려 할 때 비활성화 처리되고 "이미 구매한 질문이에요" 툴팁이 표시되는가 — `disabled` + hover 툴팁 구현
- [x] 결제 진행 중 **탭을 닫았다가 재오픈** 시 처리 중이던 주문 상태가 올바르게 복원되거나 안내가 표시되는가 — `readPendingOrder()`로 초기 state에서 바로 복원
- [x] 결제 위젯이 열린 상태에서 **뒤로가기**를 눌렀을 때 결제 플로우가 안전하게 취소되고 이전 페이지로 이동하는가 — `popstate` 이벤트에서 `onClose()` 호출


- [ ] 일반 유저로 `/admin` 접근 시 `/admin/login`으로 리다이렉트되는가
- [ ] 관리자 로그인 후 대시보드 정상 접근 가능한가
- [ ] 관리자 기능(콘텐츠 CRUD 등)이 정상 동작하는가

**극단적 케이스**

- [ ] **비로그인 상태**에서 `/admin`, `/admin/users`, `/admin/contents` 등 모든 관리자 하위 경로에 직접 접근 시 `/admin/login`으로 리다이렉트되는가
- [ ] **일반 유저 권한(`role = 'user'`)** 계정으로 로그인된 상태에서 `/admin` URL 직접 접근 시 403 또는 리다이렉트 처리가 되는가
- [ ] 관리자 세션이 **만료된 상태**에서 관리자 API를 호출하려 할 때 401/403 응답이 내려오는가
- [ ] 관리자 권한으로 **질문 삭제** 시 이미 해당 질문을 구매한 유저의 리포트 접근에 영향이 없는가
- [ ] 관리자 콘텐츠 CRUD에서 **필수 필드를 비운 채 저장** 시 유효성 검사가 막아주는가
- [ ] 관리자 콘텐츠 입력 필드에 **마크다운 또는 HTML 태그** 입력 시 렌더링 또는 이스케이프 처리가 올바른가

#### 시나리오 7 — 에러 상태 플로우

```
AI 스트리밍 실패 시 → 에러 메시지 UI 표시
네트워크 오류 시 → 재시도 버튼 또는 안내 메시지 표시
잘못된 라우트 접근 → 404 페이지 표시
```

- [ ] AI 스트리밍 실패(5xx) 시 사용자에게 에러 안내가 표시되는가
- [ ] 없는 라우트 접근 시 Next.js `not-found.tsx` 가 렌더링되는가

**극단적 케이스**

- [ ] AI 스트리밍 **중간에 연결이 끊겼을 때** (서버 응답 중단) 부분 출력된 텍스트가 그대로 표시되고 에러 안내가 노출되는가
- [ ] AI 스트리밍 중 **사용자가 페이지를 이탈**(뒤로가기, 탭 닫기)했을 때 서버 측 스트림이 적절히 중단되는가 (메모리 누수 방지)
- [ ] AI 스트리밍 **타임아웃(30초) 직전에 응답이 시작** 됐을 때 타임아웃이 올바르게 연장되거나 완료까지 대기하는가
- [ ] 네트워크를 **오프라인으로 전환** 후 버튼 클릭 시 적절한 오프라인 에러 메시지가 표시되는가
- [ ] 서버 오류(500) 응답 시 **에러 바운더리(Error Boundary)** 가 전체 앱이 아닌 해당 컴포넌트만 Fallback UI로 대체하는가

#### 시나리오 8 — 동시성 & 중복 요청 플로우 (Concurrency Edge Cases)

- [ ] **폼 제출 버튼을 빠르게 연타** 시 동일한 API 요청이 중복 발생하지 않는가 (버튼 비활성화 또는 debounce 처리)
- [ ] **분석 시작 버튼 연타** 시 동일한 세션이 중복 생성되지 않는가
- [ ] **결제 요청 중 네트워크 일시 단절 후 재연결** 시 결제 상태가 올바르게 동기화되는가
- [ ] 같은 계정으로 **두 개의 브라우저 탭에서 동시에 로그인**을 시도할 때 세션 충돌이 발생하지 않는가
- [ ] 동일한 `orderId`를 포함한 결제 성공 콜백을 **동시에 두 번** 서버로 전송 시 중복 처리가 방지되는가 (멱등성)

#### 시나리오 9 — 입력 경계값 & 보안 플로우 (Input Boundary & Security)

- [ ] 모든 텍스트 입력 필드에 **1000자 이상의 문자열** 입력 시 UI 레이아웃이 깨지지 않고 서버가 정상 처리하는가
- [ ] 입력 필드에 **이모지**(🎉, 👩‍💻 등) 및 **한글·영문·특수문자 혼합** 입력 시 인코딩 오류 없이 저장 및 표시되는가
- [ ] URL 파라미터에 **인코딩되지 않은 특수문자** 입력 시 서버가 안전하게 처리하는가 (예: `/report/abc%00xyz`)
- [ ] 브라우저 개발자 도구로 **LocalStorage/Cookie의 세션 토큰을 삭제** 후 보호된 페이지를 유지하려 할 때 로그인 페이지로 리다이렉트되는가
- [ ] **브라우저 자동완성**으로 이전 테스트 계정 정보가 채워진 상태에서 폼 제출 시 정상 동작하는가

#### 시나리오 10 — 결제 완료 후 재접근 플로우 (Post-Payment Edge Cases)

```
결제 성공 후 /payment/success 콜백 처리 완료
→ 리포트 페이지에서 AI 스트리밍 완료
→ 이후 다양한 재접근 시도
```

- [ ] AI 스트리밍이 완료된 리포트를 **나중에 다시 방문** 시 이미 생성된 리포트가 정상 표시되는가 (재생성 없음)
- [ ] 리포트 페이지를 **여러 번 새로고침** 해도 AI 스트리밍이 재요청되지 않는가
- [ ] 구매 완료된 질문의 리포트를 **로그아웃 후 재접근** 시 비로그인 안내 또는 접근 제한이 표시되는가
- [ ] 구매 완료 후 **다른 기기/브라우저에서 동일 계정으로 로그인** 시 구매한 리포트에 정상 접근 가능한가
