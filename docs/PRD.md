# VEIL PRD (콘텐츠 소비형 구조)

---

## 1. 제품 개요 (Product Overview)

### 1.1 제품 정의

VEIL은 **콘텐츠 기반 해석 서비스**다.

사용자가 콘텐츠를 클릭하고 입력하면,  
현재 상황, 관계, 감정의 구조를 카드형 콘텐츠로 제시한다.

**핵심 포지셔닝**  
이 서비스는 질문을 파는 것이 아니라  
**결과 콘텐츠를 파는 서비스**다.

---

### 1.2 사용자 경험의 본질

**탐색형 ❌**

- 사용자가 질문을 선택한다
- 선택에 따라 다음 질문이 나온다
- 루프 반복

**소비형 ✔**

- 이미 만들어진 콘텐츠가 있다
- 사용자가 콘텐츠를 클릭한다
- 입력 후 결과를 본다
- 결과를 구매한다

---

## 2. 기술 스택 (Tech Stack)

- Web Framework: Next.js 15 (App Router, Turbopack)
- Language: TypeScript 5.7+
- Styling: Tailwind CSS v4, shadcn/ui
- Backend: Next.js API Routes + Server Actions
- Database & Auth: Supabase
- Payments: Toss Payments
- AI: Anthropic SDK (Claude Sonnet 4)
- Analytics: GA4
- E2E Test: Playwright

---

## 3. 핵심 UX 흐름 (Core UX Flow)

```
콘텐츠 클릭
  ↓
입력 (콘텐츠별 다른 방식)
  ↓
결과 생성 (6장 카드)
  ↓
무료 공개 (2장)
  ↓
잠금 카드 노출
  ↓
개별 열기 또는 전체 열기 선택
  ↓
결제 → 즉시 공개
```

## 4. 콘텐츠 (Content)

### 4.1 콘텐츠 정의

콘텐츠는 "찌르는 질문 문장"이다.

사용자의 현재 상태를 구체적으로 지목하는 문장으로,  
즉시 공감 가능해야 한다.

---

### 4.2 콘텐츠 작성 원칙

- ❌ 설명형, 넓은 주제
- ✔ 즉시 공감 가능한 구체적 문장
- ✔ 감정 상태, 관계 구조, 상황 판단을 직접 겨냥

---

### 4.3 콘텐츠 분류

#### 감정 / 관계 / 애매한 상황 콘텐츠

예시:

- "왜 나는 항상 타이밍이 어긋날까"
- "이 관계, 계속 가도 되는 걸까"
- "좋은 사람인데 왜 마음이 안 갈까"

#### 비교 / 판단 / 구조 콘텐츠

예시:

- "우리 커플, 누가 더 주도적일까"
- "지금 내가 상대를 더 좋아하는 걸까"

---

## 5. 입력 구조 (Input Structure)

### 5.1 핵심 원칙

**"콘텐츠가 입력 방식을 결정한다"**

모든 콘텐츠를 동일 입력으로 만들면:

- 어떤 건 과하고
- 어떤 건 부족함
- 결과적으로 UX가 무너짐

---

### 5.2 입력 방식 - Type A (기본)

**자유입력 + 보정질문**

대상: 감정, 관계, 애매한 상황

구조:

1. 자유입력 1회 (사용자가 상황 설명)
2. 보정질문 3~5개 (해석 정제용)

목적:

- 사용자의 현재 상태를 자유롭게 드러냄
- AI가 해석할 충분한 맥락 확보
- 주관적 상황을 구조화함

---

### 5.3 입력 방식 - Type B (보조)

**선택형 질문**

대상: 비교, 판단, 구조 질문

구조:

- 8~12개 선택형 질문
- 복수 선택 가능
- 선택 패턴 기반 분석

목적:

- 명확한 구조 파악
- 선택 패턴 조합으로 해석 생성
- 빠른 입력 완료

---

### 5.4 입력의 역할

입력은 정보를 모으기 위한 것이 아니라,  
**결과 해석의 정확도를 높이기 위한 최소 구조**다.

AI가 사용자의 현재 상태를 정확히 파악하고  
맞춤형 카드를 생성하기 위해 필요한 최소한의 맥락만 수집한다.

---

## 6. 결과 구조 (Card-Based Output)

### 6.1 결과 형식

기존 리포트 형식 ❌  
**카드 단위 분리 형식 ✔**

---

### 6.2 카드 구성 (고정)

**총 6장 구성**

#### 무료 영역 (2장)

1. **핵심 한 줄** - 상황의 가장 본질적인 해석
2. **상황 재정의** - 지금 무슨 일이 일어나고 있는가

#### 유료 영역 (4장)

3. **이 관계에서 누가 주도권을 쥐고 있는지** - 관계 또는 상황의 실제 구조와 주도권
4. **너의 패턴** - 사용자 본인의 선택/반응 패턴
5. **상대 패턴** - 상대방의 행동 패턴 (있을 경우)
6. **계속 가면 어떻게 되는지** - 지금 흐름의 방향과 결과

---

### 6.3 카드 설계 원칙

- 각 카드는 **하나의 의미**만 전달
- 문장은 **짧고 단정형**
- 감정보다 **구조 설명** 우선
- 카드 간 내용은 **반드시 이어져야 함**
- 밀도로 설득 (길이 ❌)

---

### 6.4 카드 정보 기준 (절대 변경 금지)

```
무료: 2장
유료: 4장
---
절대 늘리지 마
절대 줄이지 마
```

---

## 7. 결과 페이지 UI (Result Page)

### 7.1 페이지 구성

1. **무료 카드 2장** - 즉시 표시
2. **잠금 카드 리스트** - 제목 + 미리보기 일부 + 열기 버튼
3. **선택형 열기** - 개별 카드 구매 (카드당 900원)
4. **전체 열기** - 전체 카드 일괄 구매 (2,900원)

---

### 7.2 잠금 카드 UI 요소

각 잠금 카드에 표시:

- 카드 제목 (완전 노출)
- 미리보기 문장 (처음 1~2줄)
- 개별 열기 버튼

---

### 7.3 결제 이후 UX

- 결제 즉시 카드 내용 공개
- 페이지 새로고침 없음
- **결제한 카드로 자동 스크롤 이동**
- 동일 위치에서 바로 확인 가능

---

## 8. UI 스타일 및 비주얼 방향 (Visual Direction)

VEIL의 UI는 일반적인 앱 UI가 아니다.

이 서비스는 기능 중심 UI가 아니라,
감정과 분위기를 전달하는 “비주얼 콘텐츠 UI”다.

---

### 8.1 핵심 컨셉

- UI는 “앱”이 아니라 “포스터”처럼 보여야 한다
- 카드 하나하나가 콘텐츠이자 비주얼이다
- 텍스트 자체가 디자인이 된다

---

### 8.2 디자인 키워드

- 레트로팝 (Retro Pop)
- 시티팝 (City Pop)
- 80s 감성
- 퍼플 / 핑크 / 네온 그라데이션
- 부드러운 빛 번짐 (Glow)
- 필름 노이즈 (Grain)

---

### 8.3 카드 디자인 원칙

- 카드 = UI 요소가 아니라 “콘텐츠 이미지”
- 텍스트 중심 구성 (이미지 최소화)
- 여백을 넓게 사용
- 정렬은 완벽하게 맞추지 않고 자연스럽게 흐르게
- 감정보다 분위기를 먼저 전달

---

### 8.4 금지 사항

- 일반적인 카드 UI (흰 배경, 그림자, 라운드 박스)
- 스타트업 스타일 미니멀 UI
- 과도하게 정돈된 레이아웃
- 기능 중심 디자인

---

### 8.5 핵심 한 줄

“UI를 만들지 말고, 앨범 커버를 만든다”

### 8.6 UI 구현 가이드 (Implementation Guide)

#### 8.6.1 컬러 시스템

Primary Background:

- #2D3250 (Deep Purple Base)

Gradient:

- Purple (#2D3250) → Pink (#C97B84) → Soft Orange (#F7A278)

Text:

- Main: #FAF8F5 (Warm White)
- Sub: rgba(250,248,245,0.6)

Accent:

- Neon Pink 계열 (Hover / 강조용)

---

#### 8.6.2 카드 레이아웃

- Padding: 24px ~ 32px
- Border Radius: 0px (또는 매우 낮게)
- Shadow: 사용하지 않음

Spacing:

- 카드 간 간격: 20~28px
- 내부 줄 간격: 1.4~1.6

텍스트 정렬:

- 중앙 정렬 또는 약간 비대칭 허용

---

#### 8.6.3 타이포그래피

폰트:

- 둥글고 부드러운 고딕 계열

핵심 한 줄:

- font-size: 20~24px
- line-height: 1.4

본문:

- font-size: 16~18px
- line-height: 1.6

원칙:

- 문장은 짧게 끊어서 여러 줄 사용
- 과도한 정렬 금지
- 타이포 자체가 디자인 역할을 수행

---

#### 8.6.4 효과 (Effect)

필수 효과:

- Grain (노이즈) 오버레이
- Glow (빛 번짐)
- Gradient Background

금지:

- Drop Shadow
- Glassmorphism
- 과도한 Blur

---

#### 8.6.5 잠금 카드 처리

- content 일부 blur 처리
- opacity: 60~70%
- preview 텍스트는 노출
- lock 아이콘 최소화 (텍스트 중심)

---

#### 8.6.6 인터랙션

- hover: 색상 변화 (네온 강조)
- transition: 200~300ms
- 애니메이션 최소화 (부드럽게만)

---

#### 8.6.7 핵심 원칙

- UI는 기능을 보여주기 위한 것이 아니라 분위기를 전달하기 위한 것이다
- 카드 하나하나가 콘텐츠이자 비주얼이다
- 정답은 “앱 UI”가 아니라 “앨범 커버 같은 화면”이다

## 9. 모바일 우선 설계 (Mobile-First Design)

VEIL은 모바일 환경을 기준으로 설계된다.

이 서비스의 주요 사용 맥락은 다음과 같다:

- 혼자 있을 때
- 감정적으로 흔들릴 때
- 빠르게 콘텐츠를 소비할 때

따라서 모든 UI/UX는 모바일을 기준으로 최적화되어야 한다.

---

### 9.1 설계 원칙

- Desktop은 Mobile의 확장일 뿐, 기준이 아니다
- 모든 화면은 Mobile 기준으로 먼저 설계한다
- 한 손 사용을 고려한 인터랙션
- 스크롤 기반 소비 구조 유지

---

### 9.2 레이아웃 기준

- 최대 width: 480px
- 중앙 정렬
- 좌우 padding: 16px ~ 20px

구조:

- 세로 스크롤 중심
- 카드 단위 콘텐츠 흐름 유지
- 한 화면에 하나의 감정 집중

---

### 9.3 UX 원칙

- 터치 영역 최소 44px 이상
- 입력 단계 최소화 (빠르게 끝나야 함)
- 결과는 즉시 보여줘야 함
- 로딩은 감성 메시지로 대체

예시:

- "흐름을 읽는 중이야"
- "패턴을 보고 있어"

---

### 9.4 인터랙션

- 스와이프보다 스크롤 우선
- 탭 중심 UX
- 복잡한 제스처 금지

---

### 9.5 Desktop 대응

- Mobile UI를 그대로 확장
- 카드 width만 증가
- 레이아웃 구조 변경 금지

---

### 9.6 핵심 한 줄

“이 서비스는 모바일에서 완성된다”

## 11. 분석 엔진 (Insight Engine)

### 11.1 핵심 원리 (기존 유지)

- 해석은 **사용자 입력 기반**으로 생성
- 단일 응답이 아닌 **패턴 조합** 분석
- 개인이 아닌 **관계 구조** 중심으로 해석
- 현재 상태의 구조적 재정의가 목표

---

### 11.2 해석 생성 프로세스

1. **입력 수신** - Type A (자유+보정) 또는 Type B (선택형)
2. **패턴 추출** - 선택, 반응, 표현에서 패턴 도출
3. **구조 해석** - 관계/상황의 실제 구조 파악
4. **카드 생성** - 정해진 6개 카드 프레임에 맞춰 결과 생성

---

### 11.3 출력 구조

- AI는 자유롭게 글을 생성하지 않음
- **정해진 카드 프레임**에 맞춰 생성
- 각 카드의 역할 고정
- 밀도 있는 문장으로 채움

---

## 11.4 카드 생성 로직 (Card Generation Logic)

### 11.4.1 기본 원칙

VEIL의 결과는 자유 형식 리포트가 아니라, 고정된 6개 카드 구조로 생성된다.

AI는 사용자의 입력을 바탕으로 전체 해석을 먼저 구성한 뒤,  
각 카드의 역할에 맞게 내용을 분리하여 출력한다.

카드는 서로 독립적인 정보처럼 보이지만, 전체적으로는 하나의 흐름을 가져야 한다.

흐름:

1. 핵심 한 줄
2. 상황 재정의
3. 이 관계에서 누가 주도권을 쥐고 있는지
4. 너의 패턴
5. 상대 패턴
6. 계속 가면 어떻게 되는지

---

### 11.4.2 카드별 생성 기준

#### 카드 1. 핵심 한 줄

**역할**: 사용자가 처음 보고 "이거 내 얘기다"라고 느끼게 만드는 카드

**생성 기준:**

- 상황의 본질을 한 문장으로 압축한다
- 설명보다 직관적인 진단에 가깝게 쓴다
- 너무 친절하게 풀지 않는다
- 사용자의 입력에서 가장 강하게 드러난 감정/관계 구조를 반영한다

**문장 기준:**

- 1~2문장
- 짧고 강하게
- 단정형 사용

**예시:**

- 이 관계는 애정보다 주도권 싸움에 더 가까워.
- 너는 마음이 식은 게 아니라, 확신을 잃은 상태야.
- 지금 문제는 타이밍이 아니라, 둘의 속도 차이야.

---

#### 카드 2. 상황 재정의

**역할**: 사용자가 자신의 상황을 다르게 보게 만드는 카드

**생성 기준:**

- 사용자가 스스로 해석하던 방식을 한 번 뒤집는다
- "문제처럼 보였던 것"의 실제 구조를 설명한다
- 카드 1의 핵심 한 줄을 2~4문장으로 확장한다
- 아직 유료 카드의 핵심 내용은 모두 공개하지 않는다

**문장 기준:**

- 2~4문장
- 무료 영역이지만 충분히 납득돼야 함
- 결론을 완전히 닫지 않음

**예시:**

- 네가 불안한 이유는 상대가 애매해서만은 아니야. 상대의 반응을 기준으로 네 마음을 계속 확인하고 있기 때문이야. 그래서 이 관계는 감정보다 반응의 균형이 더 크게 작동하고 있어.

---

#### 카드 3. 이 관계에서 누가 주도권을 쥐고 있는지

**역할**: 유료 결제를 가장 강하게 유도하는 핵심 카드

**생성 기준:**

- 관계나 상황 안에서 실제로 흐름을 움직이는 쪽을 해석한다
- 단순히 "누가 더 좋아하냐"가 아니라, 누가 기다리게 만들고 누가 반응하게 만드는지 본다
- 주도권은 감정의 크기가 아니라 선택권, 반응 속도, 거리 조절 능력으로 판단한다

**문장 기준:**

- 3~5문장
- 관계 구조 중심
- 자극적이되 점술/예언처럼 쓰지 않음

**예시 방향:**

- 겉으로는 상대가 더 여유 있어 보이지만, 실제 주도권은 네 반응 방식에서 생기고 있어.
- 이 관계에서 주도권은 더 좋아하는 사람이 아니라, 덜 흔들리는 쪽에 있어.
- 지금은 상대가 결정권을 가진 것처럼 보이지만, 흐름을 계속 유지시키는 건 너의 기다림이야.

---

#### 카드 4. 너의 패턴

**역할**: 사용자의 선택/반응 구조를 드러내는 카드

**생성 기준:**

- 사용자가 반복적으로 보이는 선택 방식, 감정 반응, 회피/직면 방식을 해석한다
- 성격 일반화가 아니라 "이 상황에서 작동하는 패턴"으로 설명한다
- 사용자가 스스로 인식하지 못한 반복 구조를 드러낸다

**문장 기준:**

- 3~5문장
- "넌 원래 이런 사람이야" 금지
- "이 상황에서 너는 이렇게 움직이고 있어" 방식 사용

**예시 방향:**

- 너는 확신이 없을 때 바로 움직이는 쪽이 아니야. 먼저 상대 반응을 보고, 그 반응에 맞춰 네 마음을 조정하려고 해. 그래서 선택이 늦어지는 게 아니라, 확신을 외부에서 확인하려는 구조가 생겨.

---

#### 카드 5. 상대 패턴

**역할**: 상대방 또는 상황의 반응 구조를 해석하는 카드

**생성 기준:**

- 상대가 있는 콘텐츠에서는 상대의 행동 패턴을 분석한다
- 상대가 없는 감정/진로 콘텐츠에서는 "상황 패턴"으로 대체한다
- 상대의 마음을 단정 예측하지 않는다
- 관찰 가능한 행동과 흐름을 기반으로 해석한다

**문장 기준:**

- 3~5문장
- "그 사람은 널 좋아해/안 좋아해"처럼 단정 금지
- "그 사람은 이런 방식으로 관계를 조절하고 있어"처럼 구조 설명

**예시 방향:**

- 그 사람은 감정이 없어서 애매한 게 아니라, 관계가 선명해지는 순간 속도를 늦추는 쪽이야.
- 상대는 확신을 말로 주기보다 행동의 간격으로 조절하고 있어.
- 이 상황은 네가 더 예민해서 생긴 게 아니라, 상대가 분명한 신호를 늦게 주는 구조에서 생긴 거야.

---

#### 카드 6. 계속 가면 어떻게 되는지

**역할**: 현재 구조가 유지될 경우의 흐름을 보여주는 마무리 카드

**생성 기준:**

- 미래 예언이 아니라 현재 패턴이 유지될 때의 자연스러운 흐름을 제시한다
- 사용자가 지금 무엇을 놓치고 있는지 한 문장으로 정리한다
- 해결책을 직접 제시하지 않는다
- 판단 여지를 남기되, 긴장감 있게 마무리한다

**문장 기준:**

- 3~5문장
- 예언 금지
- 조언 금지
- 마지막 문장은 강하게 남긴다

**예시 방향:**

- 이 흐름이 계속되면 관계가 끝나는 것보다 먼저, 네 기준이 흐려질 가능성이 커. 상대를 더 알게 되는 게 아니라, 상대 반응에 맞춰 너를 계속 조정하게 돼. 이 관계에서 진짜 봐야 할 건 상대의 마음이 아니라, 네가 어디까지 기다리게 되는지야.

---

### 11.4.3 카드 생성 시 금지 사항

- 카드마다 같은 말을 반복하지 않는다
- 무료 카드에서 유료 카드의 핵심 결론을 모두 공개하지 않는다
- 유료 카드가 단순히 무료 카드의 긴 설명처럼 느껴지면 안 된다
- 미래를 단정하지 않는다
- 해결책을 직접 제시하지 않는다
- MBTI, 애착유형 같은 외부 성격 프레임으로 단정하지 않는다

---

## 11.5 질문 설계 기준 (Input Question Design)

### 11.5.1 기본 원칙

질문은 검사가 아니라, 사용자의 현재 상태를 드러내기 위한 장치다.

질문은 사용자가 "정답을 고르는 느낌"이 아니라,  
"내 상태에 가까운 걸 고르는 느낌"이어야 한다.

---

### 11.5.2 Type A: 자유입력 + 보정질문 설계 기준

Type A는 감정, 관계, 애매한 상황 콘텐츠에 사용한다.

구조:

1. 자유입력 1회
2. 보정질문 3~5개

#### 자유입력 질문 기준

자유입력은 길게 쓰게 만들지 않는다.  
사용자가 부담 없이 상황을 적을 수 있어야 한다.

**좋은 예시:**

- 지금 상황을 한두 문장으로 적어줘.
- 제일 신경 쓰이는 부분만 적어줘.
- 상대나 상황에서 가장 헷갈리는 지점을 적어줘.

**나쁜 예시:**

- 지금까지의 모든 상황을 자세히 설명해줘.
- 관계의 시작부터 현재까지 서술해줘.
- 네 감정을 구체적으로 분석해서 적어줘.

---

### 11.5.3 Type A 보정질문 기준

보정질문은 자유입력의 맥락을 정리하기 위한 질문이다.

보정질문은 다음 중 최소 3개 축을 포함해야 한다.

| 축          | 목적                        | 예시                                           |
| ----------- | --------------------------- | ---------------------------------------------- |
| 현재 상태   | 상황의 종류 파악            | 썸 / 연애 중 / 이별 / 재회                     |
| 감정        | 사용자의 체감 파악          | 불안 / 답답함 / 식은 느낌 / 기대               |
| 상대 태도   | 상대의 반응 구조 파악       | 애매함 / 거리 둠 / 적극적 / 반복적             |
| 사용자 반응 | 사용자의 행동 패턴 파악     | 기다림 / 확인함 / 피함 / 정리하려 함           |
| 반복 구조   | 비슷한 일이 반복되는지 파악 | 매번 비슷함 / 이번이 처음 / 특정 상황에서 반복 |

**보정질문 작성 원칙:**

- 질문 수는 3~5개
- 선택지는 4~6개
- 복수 선택 가능 여부는 질문 성격에 따라 결정
- 선택지는 서로 겹치면 안 됨
- "기타"는 가능하지만, 핵심 흐름은 선택지만으로도 작동해야 함

---

### 11.5.4 Type B: 선택형 질문 설계 기준

Type B는 비교, 판단, 구조 콘텐츠에 사용한다.

**예시 콘텐츠:**

- 우리 커플, 누가 더 주도적일까
- 지금 내가 상대를 더 좋아하는 걸까
- 이 관계에서 누가 더 흔들리고 있을까

**구조:**

- 8~12개 선택형 질문
- 단일 선택 또는 복수 선택
- 선택 패턴 기반으로 결과 생성

Type B 질문은 반드시 서로 다른 축을 측정해야 한다.

| 축        | 목적                          |
| --------- | ----------------------------- |
| 감정 크기 | 누가 더 감정적으로 흔들리는지 |
| 반응 속도 | 누가 더 빨리 반응하는지       |
| 주도권    | 누가 관계 흐름을 조절하는지   |
| 확인 욕구 | 누가 더 확신을 필요로 하는지  |
| 거리 조절 | 누가 더 쉽게 멀어지는지       |
| 반복 패턴 | 비슷한 구조가 반복되는지      |

---

### 11.5.5 좋은 질문 기준

좋은 질문은 다음 조건을 만족해야 한다.

- 하나의 질문이 최소 2개 이상의 해석 축에 기여한다
- 선택지가 서로 다른 방향을 가리킨다
- 검사처럼 딱딱하지 않다
- 사용자가 쉽게 고를 수 있다
- 선택 결과가 카드 생성에 직접 반영된다

**좋은 예시:**

질문: 애매해지면 너는 보통 어떻게 해?

선택지:

- 먼저 확인하려고 해
- 기다리면서 상대 반응을 봐
- 괜찮은 척하면서 거리를 둬
- 마음을 접으려고 해
- 더 신경 쓰이지만 티를 안 내

**나쁜 예시:**

질문: 너는 불안형이야?

선택지:

- 맞아
- 아니야

이유:

- 사용자가 자기 유형을 직접 판단하게 만듦
- 해석 밀도가 낮음
- 결과 개인화에 도움이 적음

---

### 11.5.6 질문 설계 금지 사항

- 심리검사처럼 느껴지는 질문 금지
- "너는 어떤 유형이야?" 같은 직접 유형 질문 금지
- 점술/운세처럼 느껴지는 질문 금지
- 너무 추상적인 질문 금지
- 선택지가 의미상 겹치는 구성 금지
- 모든 콘텐츠에 동일 질문 세트 반복 금지

---

## 11.6 톤 & 언어 시스템 (Tone & Language System)

### 11.6.1 기본 톤

VEIL의 모든 결과 카드는 반말로 작성한다.

톤의 핵심은 다음 세 가지다.

| 요소        | 설명                                     |
| ----------- | ---------------------------------------- |
| 직설성      | 애매하게 돌려 말하지 않음                |
| 구조성      | 감정보다 패턴과 관계 구조를 설명         |
| 절제된 감정 | 과도한 위로 없이 필요한 만큼만 감정 반영 |

---

### 11.6.2 문장 스타일

문장은 짧고 단정적으로 쓴다.

**원칙:**

- 한 문장에는 하나의 의미만 담는다
- 긴 문장보다 짧은 문장을 여러 개 사용한다
- "~인 것 같아", "~일 수도 있어"를 남발하지 않는다
- 분석 결과는 단정형으로 쓴다
- 단, 미래는 단정하지 않는다

**좋은 예시:**

- 너는 확신이 없을 때 바로 움직이지 않아.
- 먼저 상대 반응을 보고, 그다음에 네 마음을 정리하려고 해.
- 그래서 이 관계에서는 기다림이 반복돼.

**나쁜 예시:**

- 현재 사용자는 상대방의 반응에 따라 자신의 감정을 조정하는 경향이 있는 것으로 보입니다.
- 아마도 불안한 감정이 어느 정도 작용하고 있을 수 있습니다.

---

### 11.6.3 호칭 규칙

콘텐츠의 해석 대상에 따라 호칭을 다르게 사용한다.

| 분석 대상   | 호칭                | 예시                                                    |
| ----------- | ------------------- | ------------------------------------------------------- |
| 사용자 중심 | 너 / 넌             | 너는 이런 상황에서 확신을 기다리는 쪽이야.              |
| 상대 중심   | 그 사람 / 상대      | 그 사람은 관계가 선명해지는 순간 속도를 늦추는 쪽이야.  |
| 관계 중심   | 둘 사이 / 이 관계   | 둘 사이에서 반복되는 건 감정 차이가 아니라 속도 차이야. |
| 상황 중심   | 이 상황 / 지금 흐름 | 이 상황은 선택을 미룰수록 더 복잡해지는 구조야.         |

---

### 11.6.4 금지 표현

다음 표현은 사용하지 않는다.

#### 과도한 공감

- 많이 힘들었지?
- 너무 속상했겠다
- 네 마음 다 이해해

#### 조언형 표현

- 이렇게 해보는 게 좋아
- 지금은 연락하지 마
- 이 관계는 정리하는 게 맞아

#### 예언형 표현

- 곧 좋은 일이 생길 거야
- 이 관계는 반드시 끝나
- 상대는 결국 돌아올 거야

#### 성격 일반화

- 넌 원래 이런 사람이야
- 너는 불안형이야
- 너는 회피형 인간이야

#### 점술 느낌 표현

- 운명적으로
- 인연이 아니야
- 기운이 맞지 않아
- 흐름상 정해져 있어

---

### 11.6.5 권장 표현

다음 표현은 VEIL 톤에 적합하다.

- 이 관계에서 반복되는 건...
- 지금 구조는...
- 너는 이 상황에서...
- 상대는 이런 방식으로...
- 문제는 감정이 아니라...
- 겉으로는 A처럼 보이지만, 실제로는 B에 가까워.
- 이 흐름이 계속되면...
- 여기서 놓치고 있는 건...

---

### 11.6.6 무료 카드와 유료 카드의 톤 차이

무료 카드는 빠르게 납득시키는 역할이다.  
유료 카드는 납득 이후 더 깊은 구조를 보여주는 역할이다.

| 구분      | 목적             | 톤                      |
| --------- | ---------------- | ----------------------- |
| 무료 카드 | 공감과 납득      | 짧고 강함               |
| 유료 카드 | 구조 해석과 몰입 | 더 구체적이고 밀도 있음 |

무료 카드에서는 모든 결론을 말하지 않는다.  
유료 카드에서 더 깊은 구조가 열려야 한다.

---

### 11.6.7 UI 카피 톤

UI 카피도 결과 카드와 동일하게 반말을 사용한다.

| 상황         | 문구                         |
| ------------ | ---------------------------- |
| 입력 시작    | 지금 상황을 적어줘           |
| 다음 단계    | 계속해                       |
| 결과 생성 중 | 흐름을 읽는 중이야           |
| 잠금 카드    | 이 부분은 아직 잠겨 있어     |
| 개별 열기    | 이 카드 열어보기             |
| 전체 열기    | 전체 열기                    |
| 결제 완료    | 열렸어. 바로 확인해봐        |
| 오류         | 문제가 생겼어. 다시 시도해줘 |

---

### 11.6.8 톤 최종 원칙

VEIL은 다정하게 위로하는 서비스가 아니다.  
사용자의 상황을 구조적으로 짚어주는 서비스다.

따라서 말투는 친근하지만, 내용은 흐리지 않는다.

목표는 "위로받았다"가 아니라  
"들켰다"에 가까운 감각이다.

## 11.7 사용자 타입 추론 (Internal User Inference)

### 11.7.1 목적

사용자 타입 추론은 결과에 직접 노출하기 위한 것이 아니라,  
카드 생성의 해석 방향과 문장 톤을 조절하기 위한 내부 보정값이다.

VEIL은 사용자를 특정 유형으로 단정하지 않는다.  
대신 입력 응답에서 드러나는 선택 방식, 감정 반응, 관계 인식 방식을 추론하여 카드 생성에 반영한다.

---

### 11.7.2 추론 축

| 축             | 값                                              | 설명                                                                |
| -------------- | ----------------------------------------------- | ------------------------------------------------------------------- |
| thinking_style | cognitive / emotional                           | 구조와 이유 중심인지, 감정과 체감 중심인지                          |
| reaction_style | waiting / confronting / avoidant                | 불확실할 때 기다리는지, 확인하는지, 거리를 두는지                   |
| stability_need | high / medium / low                             | 관계나 상황에서 안정감을 얼마나 필요로 하는지                       |
| focus_target   | self / other / relationship                     | 해석 관심이 나, 상대, 관계 구조 중 어디에 가까운지                  |
| decision_mode  | certainty_seeking / flow_based / distance_based | 확신을 원해서 움직이는지, 흐름을 따라가는지, 거리 조절로 판단하는지 |

---

### 11.7.3 입력 방식별 추론

#### Type A: 자유입력 + 보정질문

- 자유입력의 표현, 감정 단어, 상황 설명을 반영한다.
- 보정질문의 선택 패턴으로 추론값을 보정한다.
- 감정과 맥락이 반영되므로 문장 개인화 강도가 높다.

#### Type B: 선택형 질문

- 자유입력 없이 선택 패턴만으로 추론한다.
- 각 선택지는 1개 이상의 추론 축에 가중치를 가진다.
- 감정보다 구조 중심 해석에 적합하다.
- Type B 콘텐츠는 최소 8개 이상의 질문으로 5개 추론 축을 모두 채울 수 있어야 한다.

---

### 11.7.4 적용 방식

사용자 타입 추론은 다음 영역에 반영된다.

- 카드 1의 핵심 한 줄 방향
- 카드 2의 상황 재정의 관점
- 카드 4의 사용자 패턴 해석
- 카드 6의 마무리 긴장감
- 문장 톤의 강도 조절

---

### 11.7.5 금지 사항

- 사용자에게 타입명을 직접 노출하지 않는다.
- MBTI, 애착유형처럼 고정된 성격 라벨로 표현하지 않는다.
- "너는 원래 이런 사람"이라고 쓰지 않는다.
- 타입은 정체성이 아니라 현재 상황에서 드러난 반응 방식으로만 사용한다.

## 12. 데이터 구조 개요 (Database Overview)

### 12.1 개념

**카드 기반**

```
콘텐츠 → 입력 → 카드 결과
Users
  ↓
Contents (콘텐츠 정의)
  ↓
Responses (사용자 입력)
  ↓
Card_Results (6개 카드 결과)
  ↓
unlocked_cards (결제 후 공개 카드 추적)
```

**card_results 테이블 스키마**

```
card_results
├── id (PK)
├── session_id (FK → Sessions)
├── card_index (1~6)
├── title
├── preview (미리보기 문장)
├── content (전체 내용)
├── is_free (boolean)
├── created_at
└── updated_at
```

---

### 12.2 핵심 변경

- "질문을 산다" ❌
- "카드(결과)를 연다" ✔

결제 대상이 변함:

- 기존: 질문 선택 결제
- 변경: 카드 공개 결제

**Orders 테이블 필드 추가**

```
Orders
├── id (PK)
├── session_id (FK)
├── type ('individual' | 'full')
├── amount (결제 금액)
├── status ('pending' | 'completed' | 'failed' | 'refunded')
├── created_at
└── updated_at
```

**type 필드 역할:**

- 'individual': 개별 카드 구매
- 'full': 전체 카드 일괄 구매
- UI 처리 단순화
- 분석 및 통계 추적 용이

---

### 12.2-1 필수 추가 테이블: unlocked_cards

개별 구매 / 전체 구매 / 재접속 상황에서 열린 카드 추적용

```
unlocked_cards
├── id (PK)
├── session_id (FK → Sessions)
├── order_id (FK → Orders)
├── card_index (3, 4, 5, 6)
├── unlocked_type ('individual' | 'full')
├── unlocked_at (timestamp)
└── updated_at (timestamp)
```

**역할:**

- 개별 구매 시 해당 카드만 열림
- 전체 구매 시 3~6번 모두 열림
- 재접속 시 열린 카드 상태 복구
- 중복 구매 방지
- order_id로 결제 이력 추적 및 환불 처리
- 데이터 꼬임 방지

---

### 12.3 기존 인프라 유지

다음은 **그대로 유지**:

- **Supabase** (인증, 세션, 데이터 저장)
- **Toss Payments** (결제 플로우)
- **비회원 구조** (세션 기반)
- **분석 엔진 로직** (패턴 해석)
- **보안 정책** (동일 유지)

인증/세션 구조는 유지하되,  
결제 대상과 구매 검증 로직은 카드 기준으로 수정된다.

- 가입하면 이 결과를 마이페이지에 저장할 수 있어

회원가입 완료 시:

- 기존 session_id를 회원 계정에 연결

- 회원 마이페이지에서 영구 보관

## 20. 에러 및 엣지케이스 처리

### 20.1 결과 생성 실패

- 안내 문구: 결과를 만드는 중 문제가 생겼어. 다시 시도해줘.
- 재시도 버튼 제공
- 관리자 알림 전송

---

### 20.2 결제 완료 후 카드 공개 실패

- 결제는 완료됐지만 카드 공개에 문제가 생겼다고 안내
- 관리자 알림 전송
- 사용자는 다시 시도 가능
- 주문 상태는 completed 유지
- unlocked_cards 생성 실패 시 재처리 가능해야 함

---

### 20.3 잘못된 session_id 접근

- 404 페이지 표시
- 문구: 해당 결과를 찾을 수 없어

---

### 20.4 비회원 인증 실패

- 문구: 일치하는 기록을 찾을 수 없어
- 3회 실패 시 1분 제한

---

### 20.5 이미 구매한 카드 재구매 방지

- 이미 열린 카드는 구매 버튼 비활성화
- 전체 열기 구매 시 이미 열린 카드는 제외하지 않고, 전체 구매 상태로 통합 처리

## 22. 보안 정책 (Security)

### 22.1 서버 사이드 검증

- 모든 결제 검증은 서버에서 수행
- unlocked_cards 생성은 결제 완료 검증 후에만 가능
- 클라이언트에서 전달한 amount, card_index, session_id는 서버에서 재검증

---

### 22.2 비회원 보안

- 전화번호는 암호화 저장
- 비밀번호 4자리는 bcrypt 해시 저장
- 비회원 조회 실패 횟수 제한
- 유료 카드 접근은 unlocked_cards 기준으로 검증

---

### 22.3 관리자 보안

- 관리자 페이지는 role 기반 접근 제한
- 관리자 API는 서버에서 role 검증
- 일반 회원 접근 차단

### 24.1 콘텐츠 (Contents)

### 24.2 세션 및 입력 (Sessions)

- `POST /api/sessions`  
  콘텐츠 입력 제출 및 세션 생성

**처리:**

1. content_id 검증
2. 입력값 저장 (content_responses 테이블)
3. user_inference 생성
4. card_results 6개 생성
5. session_id 반환

- `GET /api/sessions/[sessionId]`  
  세션 상태 조회

### 24.4 결제 (Payments & Orders)

- `POST /api/orders`  
  주문 생성 (개별/전체 구매)

**요청 포맷:**

개별 구매:

```json
{
  "session_id": "sess-abc123",
  "type": "individual",
  "card_index": 3
}
```

전체 구매:

```json
{
  "session_id": "sess-abc123",
  "type": "full"
}
```

**응답 포맷:**

```json
{
  "order_id": "order-789",
  "session_id": "sess-abc123",
  "type": "individual",
  "amount": 900,
  "status": "pending",
  "toss_order_id": "toss-order-123",
  "checkout_url": "https://checkout.toss.payments...",
  "created_at": "2024-01-15T10:10:00Z",
  "expires_at": "2024-01-15T10:25:00Z"
}
```

- `POST /api/payments/confirm`  
  Toss 결제 승인 검증

**처리:**

1. order_id로 order 조회
2. amount 일치 검증
3. Toss paymentKey 검증
4. orders.status = "completed"
5. unlocked_cards 생성 (card_index 기반)
6. 결과 페이지에서 카드 즉시 공개

- `POST /api/payments/webhook`  
  Toss 비동기 결제 상태 수신 (Toss 서버 전용)

---

### 24.5 비회원 (Guest)

- `POST /api/guest/verify`  
  전화번호 + 비밀번호로 결과 인증 (재접속 시)

**요청:**

```json
{
  "session_id": "sess-abc123",
  "phone": "01012345678",
  "password": "1234"
}
```

- `POST /api/guest/link`  
  비회원 세션을 회원 계정에 연결

**비회원 접근 규칙:**

- 무료 카드: session_id만 있으면 접근 가능
- 유료 카드: unlocked_cards 기준으로 구매 여부를 확인하고, 재접속 시 guest 인증을 요구한다.
- 결제 직후: payment success 처리 과정에서 guest_access_token을 30분간 발급한다. 이 기간에는 전화번호 + 비밀번호 재입력 없이 구매 카드가 즉시 공개된다. 재접속 또는 토큰 만료 후에는 guest verify가 필요하다.
